当前项目的真实核心不是“通用 Agent”，而是一个面向 arXiv NLP 论文库的检索与分析原型系统。

**更准确的定位**
- 本科毕设原型，主题是 arXiv NLP 论文检索助手 / RAG 系统
- 核心价值在检索链路本身，而不是让大模型包办所有能力
- 数据范围目前以 arXiv、尤其是 `cs.CL` 相关论文为主

**当前已经跑通的主线**
- 数据采集：`arXiv -> SQLite`
- 数据清洗：`SQLite -> Parquet`
- 向量化入库：`Parquet -> ChromaDB`
- 在线检索：`semantic / metadata / hybrid`
- 输出组织：`Router -> Retrieve -> Render`
- 交互入口：CLI + Streamlit 页面

**当前代码里的实际架构**
- Router：把用户输入转成结构化检索意图，而不是直接回答问题
- Retrieve：执行三种检索模式
- Renderer：把检索结果组织成中文回答
- 这更接近“带 LLM 路由和中文输出层的论文搜索系统”，不是复杂多 Agent 自治系统

**当前前后端现状**
- 后端主链路已经成型，检索系统是最成熟的部分
- 前端已经有 Streamlit 聊天界面，可查询、分页、查看结果、做摘要翻译
- 离线分析脚本和图表产物已经有不少，但还没有完整集成到统一前端看板

**与开题计划相比的现实落地**
- 混合检索、Agent 编排、增量更新、基础前端已经落地
- Spark / PySpark 作为原计划技术路线写进了材料，但当前主实现以 `pandas + SQLite + Parquet` 为主
- 原计划里较重的多 Agent 协作、高可用调度、推荐系统等，目前没有真正做成系统重点

**评估项目时应重点看**
- 主链路是否完整
- 检索效果是否稳定可演示
- 增量更新是否能支撑日常维护
- 前端是否能支撑答辩展示

**不应误判的地方**
- 不要把它当成生产级学术平台
- 不要把离线分析脚本的存在等同于前端分析系统已经完整交付
- 不要按“复杂系统工程完备度”而应按“毕设原型完成度”评价它
